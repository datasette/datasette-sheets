# datasette_sheets/ — Python Backend

Datasette plugin package. Registers routes, permissions, and hooks. All database operations use the **user's database**, not Datasette's internal DB.

## URL Structure

```
/$db/-/sheets                                    → Workbook list page
/$db/-/sheets/workbook/{wbId}                    → Spreadsheet UI
/$db/-/sheets/api/workbooks                      → list/create workbooks
/$db/-/sheets/api/workbooks/{wbId}               → get/update/delete workbook
/$db/-/sheets/api/workbooks/{wbId}/sheets/...    → sheet/cell/column/SSE/presence/data APIs
```

## Permission

Single permission: `datasette-sheets-access`. All endpoints check it. Configure via:

```bash
datasette -s permissions.datasette-sheets-access true
```

## Route Registration

Every handler uses the **full explicit regex** on the decorator, e.g.:

```python
@router.POST(
    r"/(?P<database>[^/]+)/-/sheets/api/workbooks/(?P<workbook_id>[^/]+)/sheets/(?P<sheet_id>[^/]+)/rows/delete$",
    output=DeleteRowsResponse,
)
@check_permission()
async def delete_rows(
    datasette, request, database: str, workbook_id: str, sheet_id: str,
    body: Annotated[DeleteRowsBody, Body()],
):
    ...
```

No shared `SH`/`VIEWS` prefix variables — the underlying
`datasette_plugin_router` does not method-dispatch, so same-regex
handlers collide on first-registered-wins. Disambiguate writes with
distinct URL segments (`/create`, `/update`, `/delete`, `/rows/delete`,
`/columns/delete`).

## Database Layer (`db.py` + `sql/queries.sql` + `_queries.py`)

`SheetDB` wraps a Datasette `Database` object (user's DB, not internal).
It is **orchestration only** — the SQL itself lives in
`sql/queries.sql` and is compiled into `_queries.py`
(module-level functions that take `sqlite3.Connection` as their
first arg). Any new DB operation starts in `queries.sql`, not db.py.

### The codegen pipeline

```
migrations.py                                    queries.sql
      │                                                │
      ▼                                                │
 sqlite-utils migrate                                  │
      │                                                │
      ▼                                                ▼
 ephemeral .db  ──────▶  solite-dev codegen --schema <db> <sql>
                                                │
                                                ▼
                                     sql/_queries.sql.json
                                     (checked-in IR)
                                                │
                                                ▼
                              tools/gen_queries.py (reads the IR)
                                                │
                                                ▼
                                 datasette_sheets/_queries.py
```

Commands:
- `just codegen-queries` — rebuild `_queries.sql.json` + `_queries.py` after editing `queries.sql` or `migrations.py`.
- `just check-queries-fresh` — CI gate; fails if codegen output is stale.

Requires `solite-dev` on PATH (the dev build of asg017/solite — the
PyPI `solite` release is behind on the `nullable` field in the IR
that our generator relies on).

### Writing a named query

One block per operation in `sql/queries.sql`:

```sql
-- name: listWorkbooks                -- default :rows → list[Row]
-- name: getWorkbook :row             -- single row | None
-- name: getWorkbookName :value       -- scalar | None
-- name: listWorkbookNames :list      -- list of single-column scalars
-- name: insertWorkbook               -- default Void for INSERT/UPDATE/DELETE
```

Result-type annotations are solite-dev syntax. Use them to match
what the caller actually needs; mismatches show up as TypeErrors on
unpacking.

Parameter sigils:
- `$foo::text` — typed param → `foo: str` in the generated signature.
  Known types: `text`, `integer`/`int`/`bigint`, `real`/`float`, `blob`, `boolean`.
- `:foo` — untyped → `foo: Any`. Use for nullable columns (the
  generator can't currently emit `str | None` for params) and for
  `computed_value`-style BLOB-affinity values.

The generator maps `snake_case` automatically: `listWorkbooks` →
`list_workbooks` function + `ListWorkbooksRow` dataclass.

### Idioms this codebase uses

**RETURNING skips the post-SELECT.** Both `insert*` and `update*`
helpers are `:row` with `RETURNING <full-row>`, so Python callers
get the fresh row back in one round-trip:

```sql
-- name: insertWorkbook :row
insert into datasette_sheets_workbook (id, name, created_by)
values ($workbook_id::text, $name::text, :created_by)
returning id, name, created_at, updated_at, created_by, sort_order;
```

**Partial update via sqlc-style `CASE WHEN`.** Codegen needs
fixed-shape SQL, so dynamic `SET` clauses don't work. Instead,
every mutable column takes a companion `$<col>_do_update::boolean`
flag. When False the `ELSE <col>` branch preserves the existing
value; when True the accompanying `$<col>` wins — `NULL` included,
so "set to NULL" and "leave alone" stay distinguishable.

```sql
-- name: updateWorkbook :row
update datasette_sheets_workbook
set name = case when $name_do_update::boolean then $name::text else name end,
    sort_order = case when $sort_order_do_update::boolean then $sort_order::integer else sort_order end,
    updated_at = strftime('%Y-%m-%dT%H:%M:%f', 'now')
where id = $workbook_id::text
returning id, name, created_at, updated_at, created_by, sort_order;
```

Python side packs the pair:

```python
_queries.update_workbook(
    conn,
    workbook_id=wb_id,
    name_do_update="name" in kwargs,
    name=kwargs.get("name", "") if "name" in kwargs else "",   # sentinel, ignored
    sort_order_do_update="sort_order" in kwargs,
    sort_order=kwargs.get("sort_order", 0) if "sort_order" in kwargs else 0,
)
```

**Timestamps are code, not data.** `updated_at = strftime(...)`
inline in the UPDATE — don't pass a timestamp from Python.

**Ref-rewrite UPDATEs deliberately skip `updated_at`.** The
`updateCellRaw` helper (used by `_rewrite_formulas_for_*`) does NOT
touch `updated_at`. Structural rewrites are book-keeping, not user
edits; the parent sheet's `touch_sheet` covers the change signal.

**`touch_sheet` is explicit.** Cell / named-range / shift mutations
call `_queries.touch_sheet(conn, sheet_id=...)` at the end of their
write closure. We deliberately didn't use a SQLite AFTER-INSERT
trigger — a 1000-cell paste would fire the trigger 1000 times
against the same row. If latency bites, revisit.

### What stays as direct `conn.execute`

Three sites in `db.py`, all intentional:
- `create_view` executes `view_sql.build_view_sql(spec)` + its
  triggers — **dynamic DDL** built from validated identifiers.
  Doesn't fit the named-query shape.
- `delete_view` issues `DROP VIEW IF EXISTS [{view.view_name}]` —
  same story (`view_name` is validated at create time by
  `view_sql.validate_view_name`).

Everything else — every SELECT, INSERT, UPDATE, DELETE — flows
through `_queries`.

### Load-bearing behaviours you can't see from the Python

These are documented in `sql/queries.sql`; read the comments there
before touching the affected queries.

- **`upsertCell` uses `excluded.format_json`, NOT
  `COALESCE(excluded.format_json, format_json)`.** The client sends
  `format_json = NULL` to revert a cell back to default formatting
  (e.g. unbold). COALESCE would stick with the previous JSON and
  the cell would stay bold across refresh.
- **Row/column shift is a two-pass negative-buffer UPDATE.** SQLite
  iterates UPDATE by rowid, not PK order, so a naive
  `SET row_idx = row_idx - N` crashes with scrambled-insert history
  (UNIQUE(sheet_id, row_idx, col_idx)). Pass 1 sends surviving rows
  into `-(x + 1)`; pass 2 flips back with the shift applied. Both
  passes are injective by construction, so no collision. Regression
  tests: `tests/test_row_shift.py`, `test_col_shift.py` — they
  exercise the codegened helpers directly against scrambled inserts.
- **`_rewrite_formulas_for_{deletion,insertion,move,row_move}` must
  run before the physical DELETE/shift.** The engine's
  `adjust_refs_*` expect pre-shift coordinates. Frontend mirror:
  `cells.deleteRowsLocally` / `deleteColsLocally` /
  `moveColsLocally` / `moveRowsLocally` via the matching
  `engine.adjustRefsFor*` wrapper.
- **Named-range definitions and view-registry bounds shift in
  lockstep with cell formulas** for every structural op (delete,
  insert, col-move, row-move). The trio of helpers
  (`_rewrite_formulas`, `_rewrite_named_ranges`, `_update_views`)
  share inner loops; specialise for each op via the engine
  primitive they pass through. Cell formulas use the *positional*
  variant of each move primitive (bounded ranges denote
  rectangles); named-range definitions use the *data-following*
  variant (bounded named ranges denote named cells). View bounds
  are recomputed as the bounding box of forward-mapped indices.
- **Filter rectangles also shift in lockstep** via
  `_update_filter_for_*`. Three differences from `_update_views_*`:
  (a) a filter that's fully erased gets DELETED outright (no
  registry-row debris because filters don't hold DDL); (b)
  predicate keys are stringified `col_idx` values which re-key
  under `fwd_col`; (c) `sort_col_idx` shifts under `fwd_col` and
  clears the sort if it maps to None. SSE: the `cells` route
  reads `get_filter` before + after the structural op and emits
  `filter-update` / `filter-delete` via
  `routes/helpers.py::emit_filter_change_if_any`.
- **Move ops have no `#REF!` case.** Block move is a permutation
  — no col/row is erased. Only delete produces `#REF!`.

### Coordinate helpers

Live in the engine (`lotus.cell_id(row, col)` / `parse_cell_id("A1")` /
`index_to_col(5)` / `col_to_index("F")`). Don't re-implement the A1
grammar in Python — the grammar is owned in one place
(`lotus-core`) instead of redefined per language.

### Tests

- `tests/test_gen_queries.py` — unit tests over `tools/gen_queries.py`. Compiles generated output into throwaway modules and inspects annotations + dataclass fields.
- `tests/test_queries.py` — integration tests: apply migrations to an in-memory connection, exercise each `_queries.*` function directly. Every new query added to `queries.sql` gets at least a round-trip test here.
- `tests/test_row_shift.py` / `test_col_shift.py` — regression tests for the negative-buffer invariant, now calling the codegened helpers.
- HTTP-level tests (`test_workbooks.py`, `test_cols.py`, `test_views.py`, …) exercise the full stack through the routes and flow through the codegened layer unchanged.

## SSE Endpoint

`api_events()` is a raw ASGI handler (takes `send` and `receive` params, returns `None`). It:
1. Sends `text/event-stream` headers via ASGI `send()`
2. Subscribes to the sheet's broadcast channel
3. Loops: dequeue events or send heartbeat every 30s
4. Watches for `http.disconnect` via a background `receive()` task
5. Cleans up channel subscription on disconnect

Event types: `connected`, `cell-update`, `sheet-meta`, `presence`, `heartbeat`, `rows-deleted`, `columns-deleted`.

## Presence

Ephemeral (never stored in DB). The presence endpoint resolves actor info from:
1. `request.actor` dict (e.g. `name`, `profile_picture_url` from debug-gotham)
2. `datasette-user-profiles` `get_profile()` (if installed)

Broadcasts to all other SSE clients. Actor color is deterministic via `hash(actor_id) % 10`.

## Templates

- `sheets_workbook_list.html` — server-rendered workbook list with inline `<script>` for create form.
- `sheets_base.html` — thin shell that loads Vite-built JS/CSS. Passes `data-database`, `data-workbook-id`, and `data-workbook-name` to the Svelte app via DOM attributes. Adds `body.sheets-fullscreen` so Datasette's layout gets flex-columned into a full-viewport grid (the scoped CSS lives in this template's `extra_head`).

## Engine primitives used here (from `lotus`)

All come from the Rust `lotus-core` crate via the `liblotus` PyPI
wheel (importable as `lotus`; see root `CLAUDE.md` for the version-pin
procedure):

- `Sheet().set_cells(...)` / `.get_all()` — formula evaluation + DAG recalc.
- `Sheet().set_cells_typed(...)` — kind-discriminated write path. The recalc loop calls this so per-cell typed overrides (force-text in v1) survive. Input shape: `[(cell_id, {"kind": "raw"|"string"|"number"|"boolean"|"empty"|"custom", ...})]`.
- `Sheet().register_datetime()` — opt-in; registers `jdate` / `jtime` / `jdatetime` / `jzoned` / `jspan` handlers so ISO date strings auto-classify and date arithmetic resolves to spans. Must be called on every fresh `Sheet()` (the recalc loop builds a new one per call). Built behind the `--features datetime` cargo flag (Justfile already passes it).
- `Sheet().try_parse(type_tag, raw, options)` — probe a registered handler with handler-defined options (e.g. strftime pattern for `jdate`). Returns `{type_tag, data}` or `None`. Not yet wired into the write path; reserved for future column-type hints.
- `extract_refs(formula)` — lexed references, including whole-column/whole-row ranges (`kind` discriminator).
- `parse_range(...)`, `is_unbounded_range(...)`, `parse_cell_id(...)` — canonical A1 parsing.
- `adjust_refs_for_deletion(formula, deleted_cols=..., deleted_rows=...)` — Google-Sheets-style ref rewrite on delete.
- `adjust_refs_for_insertion(formula, inserted_cols=..., inserted_rows=...)` — mirror for insert; refs at-or-past each insertion point shift outward.
- `adjust_refs_for_column_block_move(formula, src_start, src_end, final_start)` — column drag-reorder rewrite. Block move (single-col is just `src_start == src_end`) has no `#REF!` case — it's a permutation. Bounded ranges stay positional; whole-col refs use interior-bbox semantics. See `[sheet.column.drag-reorder]`.
- `adjust_refs_for_column_block_move_data_following(formula, ...)` — variant of the above where bounded ranges *also* follow via interior-bbox. Used for named-range definitions; cell formulas stick with the positional variant.
- `adjust_refs_for_row_block_move(formula, src_start, src_end, final_start)` — row-axis sibling. Same shape: positional bounded ranges, interior-bbox whole-row, whole-col unaffected.
- `adjust_refs_for_row_block_move_data_following(formula, ...)` — row-axis data-following variant for named-range definitions.
- `rewrite_refs_for_deletion(formula, ...)` — conservative variant (only inserts `#REF!`, no shifting). Not currently used.

Rule of thumb: if you're about to `import re` to look at formula or
ref text, **stop** — add a primitive to liblotus instead (see
`../TODO-liblotus-*.md` history for the pattern).
